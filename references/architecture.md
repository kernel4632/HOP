# 架构主线

## 唯一链条

一切交互与业务流程必须按以下链条组织，不允许例外：

**触发事件 → 指令执行 → 数据修改 → 效果反馈**

## 各层职责

### 触发入口

- 入口只负责接收触发、判断谁触发了什么、调用对应指令。
- 入口不处理业务逻辑，不做数据计算，不直接修改数据。
- 入口可以是 UI 组件的事件处理函数、HTTP 路由入口、CLI 命令解析器、消息监听器等。

### 指令执行

- 指令只负责完成一次清晰的业务动作：读取数据、处理逻辑、修改数据、返回结果。
- 指令不操作 UI，不发送响应，不做渲染。指令只改数据。
- 按被操作的主体命名文件并集中该主体全部操作，如 `commands/task.js` 包含 `add`、`remove`、`toggle`、`rename`。
- 指令之间通过组合调用实现复杂功能，不嵌套不继承。

### 数据存储

- 数据集中存放在 store / stores / data 中，结构显式可见。
- 数据文件只负责定义和暴露数据结构，不包含业务逻辑。
- 数据变化自动驱动 UI 更新或其他反馈（响应式机制由框架提供）。
- 数据结构必须带注释说明每个字段的含义和格式。

### 副作用响应

- 数据变化引发的自动响应逻辑（监听、订阅、钩子）集中管理在 watchers / listeners / hooks 中。
- 不允许将副作用逻辑散落在各个组件或指令中，否则追踪困难。
- 每个副作用监听必须注释说明：监听什么数据、为什么监听、触发后做什么。

### 通用工具

- 仅放多主体复用、可移植到其他项目的纯工具函数。
- 只有多个主体都在用、并且拿到别的项目也能继续用的函数才允许放进去。
- 工具函数不含业务身份，不操作 store，不调用指令。

## 项目结构规则

| 文件数量 | 组织方式 |
|----------|----------|
| 少于 4 个 | 直接平铺在目录下 |
| 4–8 个 | 按业务分 1 层文件夹 |
| 超过 8 个 | 按业务分 2 层文件夹 |

每个文件夹至少含 4 个文件，否则不建文件夹。

## 模块独立性

每个模块应当具备独立性，能够被单独理解、单独移除或单独替换，不与其他模块产生隐式耦合。模块的职责边界必须清晰，便于拔插。

判断标准：如果删除某个模块后，其他模块需要大面积修改才能编译/运行，说明耦合过紧，需要重新划分职责。

## 追踪验证

任何人看到入口的一个调用，都能顺着链条完成以下追踪：

1. 入口文件里看到事件触发 → 调用了哪个指令
2. 指令文件里看到这个方法 → 读了哪些数据、改了哪些数据
3. 数据文件里看到完整数据结构 → 知道改动影响了什么
4. 反馈自动发生（响应式 UI、返回值、日志等）

如果追踪过程中需要跳转超过 3 个文件，或者中间有隐式传递（事件总线、魔法字符串、全局变量），说明架构偏离了主线。
