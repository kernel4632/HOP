# 注释准则

## 核心定位

注释是 HOP 的核心竞争力。注释的目的不是解释难点，而是让初学者连续顺畅地阅读代码，像读教材一样理解每一步。

## 密度要求

- 代码默认配备高密度注释。每一行承载业务含义的代码都必须有注释，不论该行看起来多简单。
- 禁止"只在复杂处注释"的偷懒做法。初学者需要的是连续理解，不是跳跃式猜测。
- 一个函数内不应存在连续超过 2 行的"裸代码"（无注释代码）。如果出现，说明注释密度不足。
- 注释必须有信息量。禁止废话式注释。
- 每条注释必须回答以下至少一个问题：
  - 这步在做什么业务动作？
  - 为什么这么做？
  - 数据从哪来？
  - 会变成什么？
  - 会影响哪里？
  - 这个写法的好处是什么？

## 禁止的废话注释

以下类型的注释只是在复述语法，不提供任何信息增量，禁止出现：

```
count = 0           // 将count设为0
arr.push(item)      // 将item推入数组
for i in range(10)  // 循环10次
return result       // 返回结果
```

正确做法是说明业务含义：

```
count = 0                              // 计数器归零，准备重新统计未读消息
arr.push(item)                         // 将新任务加入待处理队列
for i in range(10)                     // 逐个检查10个传感器的状态
return result                          // 返回过滤后的有效订单列表
```

## 格式规则

### 文件头注释

使用多行注释格式，用流畅精简的大白话说清楚：
- 这个文件做什么
- 设计思想是什么
- 里面有什么重要数据或方法
- 怎么调用（给出完整带参数的调用示例）

不使用 `@module`、`@description`、`@author`、`@param`、`@returns` 等标注格式。

不同语言的多行注释语法：
- JavaScript/TypeScript/Java/C/Go：`/* ... */`
- Python：`""" ... """`
- Ruby：`=begin ... =end`
- HTML/CSS：`<!-- ... -->` / `/* ... */`

### 函数段落标题

使用段落标题格式作为每个函数或逻辑段的视觉分隔标题：

- JavaScript/TypeScript/Java/C：`// --- 动作描述 ---`
- Python/Ruby/Shell：`# --- 动作描述 ---`
- HTML：`<!-- --- 动作描述 --- -->`
- CSS：`/* --- 动作描述 --- */`

标题描述该段要完成的动作，不是复述函数名。

### 尾随注释

当一行注释对应一行代码时使用，写在代码行末尾。

对齐规则：同一代码块内的尾随注释应对齐到统一的列位置，形成"左侧代码、右侧注释"的双栏阅读体验。具体对齐位置为该代码块中最长代码行末尾再加若干空格后统一开始。

连续单行操作应紧凑排列、每行配备尾随注释，形成流程整体，不留空行。

### 行注释

当一行注释对应多行代码块时使用，写在代码块上方。

### 必须注释的位置

无论代码多"自解释"，以下位置必须有注释：

- import / require / include 语句：说明引入的是什么能力
- export / module.exports 语句：说明导出了什么
- 模块级变量声明：说明该变量的用途
- 函数参数（在函数头注释中或参数行尾随）
- 卫语句：说明为什么要提前返回

### 语义段落组织

- 同一语义段中的代码紧凑排列不留空行
- 语义段切换时空行分隔
- 每个函数之间空 2 行

## 必须注释的场景

- 魔法数字：为什么是这个值
- 特殊顺序：为什么这个操作要在那个操作前面
- 不符合直觉的写法：为什么不用更常见的方式
- 为兼容保留的代码：兼容什么、什么时候可以删
- 改了会影响别处的代码：影响哪里、为什么关联
- 算法技巧：这个写法的优势和原理
- 少见写法：这个语法/API 是做什么的
- 设计决策的取舍理由：为什么选 A 不选 B

把写代码时的思考写进注释。代码会随时间遗忘背景，注释永远保留决策上下文。
