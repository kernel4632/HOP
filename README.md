# 面向人类编程（HOP）完整规范
## 目标
让刚接触项目的新人，甚至让完全不懂编程的人，只看局部代码就能快速理解、学习并接手整个项目，包括某个文件某段代码在做什么、能改什么、能怎么改，同时从阅读代码的过程中还能学习该编程语言以及学到编程技巧和算法思路。
代码必须做到零背景知识依赖，不需要记住项目架构和历史。代码即文档，文件头就是使用手册，注释就是教学材料或操作指南，命名就是说明书。该规范追求用极少极精简的代码实现完整功能，同时保持极高的可读性和可维护性，让代码既精简又易懂。
## 架构思想
触发事件→指令执行→数据修改→效果反馈。所有交互和业务流程都必须按这个链条组织，UI 交互事件或外部调用导致的触发就分配并传递给对应执行的方法（事件驱动式的调用指令执行），指令负责具体的业务逻辑计算或修改数据（指令驱动式的方法暴露与被调用），数据变化自动改变 UI 或触发其他反馈（响应式的非业务逻辑自动处理）。这个架构的目标是让数据流动清晰可追踪，职责分离非常明确，任何人看到一个事件触发处理方法就能顺着链条找到具体调用的指令执行方法在哪、数据在哪、最终影响什么。
## 示例架构规范使用
vue 前端项目按这个架构思想设计为：components 文件夹存放页面结构和事件触发相关的组件 vue 文件；commands 文件夹按被操作的主体进行文件命名并存放该主体的所有指令操作方法；store.js 文件存放所有业务用到的全部数据结构和状态，如果数据过于庞大就按主体拆分为 stores 文件夹里多个子文件分别存放和被调用读写；assets 存放所有可以被引用的资源文件。
Python 后端项目按这个架构思想设计为：main.py 或入口文件只负责接收外部触发；actions 或 commands 文件夹按被操作的主体对象进行文件命名，比如 user.py、order.py、file.py、task.py，每个文件集中存放该主体可以被外部触发调用的所有指令方法，指令方法负责完整表达一次业务动作，比如 user.add()、order.pay()、file.remove()、task.stop()，并按读取数据、处理逻辑、修改数据、反馈/返回结果的顺序组织；store.py 或 stores 文件夹里的子文件存放程序运行时所有需要共享调用的数据等，让人类能直观显式地看到整个业务会用到的数据有哪些与数据结构长什么样，如果项目需要做持久化存储就将数据集中放在 data.py 或 data 文件夹的子文件里；tool.py 或 tools 文件夹存放真正通用、和具体业务主体无关的辅助函数，只有多个主体都会用，并且拿到别的项目也能继续用的小功能才允许放进去。整个调用链必须始终遵循触发入口→指令执行→数据修改→效果反馈，入口只判断谁触发了什么并调用哪个指令，指令只完成一次清楚的业务动作，store 或 data 只保存和提供数据，tools 只提供无业务身份的小工具，反馈/返回结果只负责把执行结果用调用者能理解的方式交回去，任何人看到入口里的一个调用，都能顺着找到对应指令方法、看到它读写了哪些数据、用了哪些辅助工具、最后知道结果如何反馈。
## 命名要求
用项目里真实存在的东西命名如用户、订单、商品、页面、任务，不用技术术语比如 controller、service、repository、model，不用抽象概念如 manager、handler、processor。保持扁平结构，文件少于 4 个直接平铺，文件 4 到 8 个按业务分 1 层文件夹，文件超过 8 个按业务分 2 层文件夹，每个文件夹至少 4 个文件否则不建文件夹。工具函数少于 5 个放在一个 tools 文件里，5 个及以上建 tools 文件夹每个大工具一个文件，工具函数必须是真正通用的业务逻辑不放这里。文件命名优先使用单个单词比如 score、student、order，依靠文件结构的上下文让人理解如 user.score，只有当文件结构上下文无法提供足够信息时，才使用两个或多个单词组合命名，比如 userScore、orderList、userAPI、parseJSON。命名统一使用缩写整体保留的语义驼峰规则，普通单词按小驼峰或大驼峰连接，约定俗成的缩写必须作为一个不可拆开的整体保留，不允许写成首字母大写、其余小写的假单词形式。缩写单独出现或放在命名开头时使用小写，比如 id、url、api、http、json、apiUser、urlList、jsonText；缩写接在普通单词后面时整体使用大写，比如 userID、fileURL、userAPI、sendHTTP、parseJSON；类名、对象名或需要明显表达专有缩写的名称中，缩写也整体大写，比如 UserAPI、HTTPServer、JSONData、SSEStream。禁止把固定缩写写成 Id、Url、Api、Http、Json、Sse 这种一大一小的形式，所以必须写 userAPI 而不是 userApi，getUserID 而不是 getUserId，parseJSON 而不是 parseJson，HTTPServer 而不是 HttpServer，SSEStream 而不是 SseStream。文件名如果采用代码式命名，就遵循同样规则，比如 userAPI.js、httpServer.py、parseJSON.js；缩写只允许使用约定俗成的固定写法，比如 id、url、api、http、json、sse、sql、html、css，禁止出现 usr、cfg、svc、repo 这种不常见缩写。方法名使用动词加名词结构比如 addStudent、getScore、removeOrder，变量名直接说是什么比如 userName、totalScore，布尔值使用 is、has、can、should 开头比如 isReady、hasPermission、canEdit。指令文件导出的对象命名直接使用主体名不加后缀，比如 commands/browser.js 导出多个方法或 Browser 而不是 BrowserCommand，调用时写成 Browser.open() 或 browser = Browser(args) 后调用 browser.open() 而不是 BrowserCommand.open()，因为文件结构"commands/"已经表明这是指令相关内容无需重复强调。
## 注释要求
文件头注释使用多行注释格式，用流畅且精简大白话直接说清楚这个文件做什么、设计的思想是什么、里面有什么重要数据或方法、可以被怎么调用、一些备注说明等，不使用标注格式比如"负责什么"、"主要数据"，而是用自然语言直接描述。调用示例必须完整且真实，给出多个带参数的调用例子，让人看一眼就知道怎么用而不需要跳过去看函数定义，比如写"browser.open(url)"而不是"open() 打开网页"。数据定义的注释写在变量声明的尾部，一一对应方便阅读，复杂数据结构要举例说明格式。函数头注释使用多行注释格式，简单函数尽量一行说清楚是做什么的、怎么用、注意事项等，复杂函数或需要示例说明的分多行写，用精简大白话不使用标注格式。注释分为行注释和尾随注释两种，行注释用于给一整块代码做说明写在代码块上方，尾随注释用于给单行代码做说明写在代码行末尾，当一行注释对应一行代码时使用尾随注释，当一行注释对应多行代码时使用行注释。连续的单行操作应该使用尾随注释并紧凑排列形成一个整体流程不留空行，只有当语义段落切换时才空行分隔。尽可能多地使用尾随注释去解释代码而不允许只靠代码自解释，要考虑小白学习代码，复杂逻辑、少见写法、算法技巧必须详细注释解释为什么这么写、这个写法的好处，让编程小白从阅读代码中学习。魔法数字、特殊顺序、不符合直觉的写法、为兼容保留的代码、改了会影响别处的代码都必须注释说明原因，把写这段代码时的思考写进代码注释里。
函数结构固定按这个顺序组织：函数头注释、卫语句、读取数据/处理逻辑/修改数据、反馈/返回结果。每条卫语句都要加注释说明原因，卫语句和主逻辑之间空一行，卫语句和主逻辑之间空一行视觉上明确区分。函数长度以 4 到 16 行为甜区，超出时优先考虑拆分，但小逻辑、小判断、小变更优先就地写保持阅读不断，只有当一整块逻辑已经形成独立语义、拆出来后更容易理解时才拆成单独函数，拆出的函数至少承载一小块完整逻辑，禁止拆成少于 3 行或只用一次的碎片函数。一次完整行为可以写成较长函数只要阅读顺序清楚、段落清楚、数据流清楚，提前返回的简单判断优先写成一行紧凑形式减少视觉噪音如`if (!nodes.length) return [];`。函数要么做业务操作逻辑要么做工具辅助函数。指令函数必须职责单一，一个指令只完成一个清楚的业务动作，如`Node.add()`，指令之间通过组合而不是嵌套来实现复杂功能，指令不直接操作 UI 而只修改数据，让数据变化驱动 UI 更新。
## 代码要求
代码按自然动作分段，不同语义段之间留空行，同一语义段中的代码紧凑排列使用尾随注释形成流程整体，相关代码紧靠在一起不相关的代码之间留空行隔开，每个函数之间空 2 行。复用规则是第一次出现的逻辑优先就地写清楚保持上下文完整可读，同样逻辑稳定出现两次及以上再提取，提取的前提是提取后更容易理解、更容易搜索、更容易修改，提取后的公共内容必须有明确名字和明确用途，提前为可能复用而抽象的函数一律不写等需求真实出现再处理。代码中必须让人快速看出数据从哪里来、经过什么处理、最后到哪里去，数据变化优先显式书写，局部代码必须尽量自带足够上下文让人只看这一小段也能知道数据在这里被读、被改、被返回、被影响，通过精简的注释包含整个项目的上下文以及开发时的思考。
生成代码时固定按这个顺序展开：先写开发计划，给项目结构列出所有文件每个文件一句话说明作用，再给功能说明一句话说这次要实现什么，然后给改动位置列出要改哪些文件每个文件改什么，开始按文件编写对应代码并包含所有注释和补全没有注释的代码的注释，标注清楚你的思考和讲清楚依赖的项目上下文，最后给使用说明告诉人类怎么运行、怎么测试、以后怎么改。整个生成过程始终围绕`触发事件→指令执行→数据修改→效果反馈`这条主线展开，让人能清晰看到如何触发事件、事件如何调用指令、指令如何计算和操作数据、数据如何驱动反馈。看项目结构能快速定位主要部分，看目录名和文件名能快速猜出作用，看文件头注释能立刻知道如何使用这个文件和代码结构是什么样的、怎么设计的，看函数注释能立刻知道这个函数要做什么，看指令操作流程能顺着读懂行为顺序，看局部代码能大致判断怎么改，看代码注释能学到编程技巧和算法思路，同一主体的操作指令要集中、直观，代码阅读过程中不会因为过多碎函数、碎文件、碎跳转而中断理解，生成结果始终优先降低理解成本、追溯成本、修改成本、上下文依赖和心智消耗，始终保持局部可读、局部可用注释拔插、局部可插入续写，同时让代码成为最好的编程教材。
