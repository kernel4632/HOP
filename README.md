# 面向人类编程（HOP）完整规范
HOP 的核心目标是让完全不懂编程的人、刚接触项目的新人、甚至编程小白，只看局部代码就能理解在做什么、能改什么、怎么改，同时从阅读代码的过程中学到编程技巧和算法思路。代码必须做到零上下文依赖，不需要记住项目历史、不需要理解整体架构、不需要作者在旁边解释。代码即文档，文件头就是使用手册，注释就是操作指南和教学材料，命名就是说明书。HOP 追求用最少的代码实现功能，同时保持极高的可读性和可维护性，让代码既精简又易懂。
HOP 遵循统一的架构思想：事件→指令→数据→反馈。所有交互和业务流程都按这个链条组织，UI 交互或外部触发产生事件，事件驱动调用指令，指令负责计算或修改数据，数据变化自动响应式改变 UI 或触发其他反馈。这个架构让数据流动清晰可追踪，职责分离明确，任何人看到一个事件处理函数就能顺着链条找到指令在哪、数据在哪、最终影响什么。前端项目按这个思想组织为：components 文件夹存放页面结构和事件触发相关的组件文件，commands 文件夹按主体命名文件存放该主体的所有指令操作，store 文件存放数据结构和状态管理如果数据庞大就按主体拆分为 stores 文件夹，assets 存放资源文件。后端或脚本项目提炼这个思想为：触发入口文件负责接收请求或事件，actions 或 commands 文件夹按主体存放业务操作指令，data 或 models 文件夹存放数据结构和数据库操作，结果通过返回值或回调反馈。核心是让事件、指令、数据三者职责清晰分离，事件只负责触发不写业务逻辑，指令只负责操作不管数据存储细节，数据只负责存储不管如何被使用，反馈通过响应式或显式调用自动完成。
项目结构按真实事物组织，用项目里真实存在的东西命名比如用户、订单、商品、页面、任务，不用技术术语比如 controller、service、repository、model，不用抽象概念比如 manager、handler、processor。保持扁平结构，文件少于 4 个直接平铺在根目录，文件 4 到 10 个按业务分 1 层文件夹，文件超过 10 个按业务分 2 层文件夹到此为止，每个文件夹至少 3 个文件否则不建文件夹。工具函数少于 3 个放在一个 tools 文件里，3 个及以上建 tools 文件夹每个工具一个文件，工具函数必须是真正通用的业务逻辑不放这里。文件命名优先使用单个单词比如 score、student、order，依靠文件结构的上下文让人理解，只有当文件结构上下文无法提供足够信息时才使用两个单词的驼峰命名比如 userScore、orderList。命名统一使用驼峰格式，文件名、函数名、变量名、类名都遵循驼峰规则，优先使用初中英语词汇比如 student、teacher、book、order、user、score、class、shop，避免使用生僻词比如 repository、infrastructure、orchestrator、facade，缩写只用约定俗成的比如 id、url、api、http、json。函数名使用动词加名词结构比如 addStudent、getScore、removeOrder，变量名直接说是什么比如 userName、totalScore，布尔值使用 is、has、can、should 开头比如 isReady、hasPermission、canEdit。指令文件导出的对象命名直接使用主体名不加后缀，比如 commands/browser.js 导出 Browser 而不是 BrowserCommand，调用时写成 Browser.open() 而不是 BrowserCommand.open()，因为文件结构已经表明这是指令相关内容无需重复强调。
文件头注释使用多行注释格式，用流畅的精简大白话直接说清楚这个文件做什么、里面有什么重要数据或方法、可以怎么调用、改了会影响什么，不使用标注格式比如"负责什么"、"主要数据"，而是用自然语言直接描述。调用示例必须完整且真实，给出 3 到 5 个带参数的调用例子，让人看一眼就知道怎么用不需要再看函数定义，比如写"Browser.open(url)"而不是"open() 打开网页"。数据定义的注释写在变量声明的上方，一一对应方便阅读，复杂数据结构要举例说明格式。函数注释使用多行注释格式，简单函数尽量一行说清楚做什么、输入输出、会改什么、结果是什么，复杂函数或需要示例说明的分多行写，用精简大白话不使用标注格式。注释分为行注释和尾随注释两种，行注释用于给一整块代码做说明写在代码块上方，尾随注释用于给单行代码做说明写在代码行末尾，当一行注释对应一行代码时使用尾随注释，当一行注释对应多行代码时使用行注释。连续的单行操作应该使用尾随注释并紧凑排列形成一个整体流程不留空行，只有当语义段落切换时才空行分隔。假设阅读者是编程小白，内行人一眼能看懂但外行人需要思考的代码必须加注释，复杂逻辑、少见写法、算法技巧必须详细注释解释为什么这么写、这个写法的好处、可以学到什么，让编程小白从阅读代码中学习。魔法数字、特殊顺序、不符合直觉的写法、为兼容保留的代码、改了会影响别处的代码都必须注释说明原因。
函数结构固定按这个顺序组织：函数注释、卫语句也就是提前返回的错误检查、读取数据、处理逻辑、修改数据、返回结果。卫语句每条之间空一行，每种情况加注释说明原因，卫语句和主逻辑之间空一行视觉上明确区分。函数长度以 4 到 10 行为甜区，超出时优先考虑拆分，但小逻辑、小判断、小变更优先就地写保持阅读不断，只有当一整块逻辑已经形成独立语义、拆出来后更容易理解时才拆成单独函数，拆出的函数至少承载一小块完整逻辑不拆成只包一两行且只用一次的碎片。一次完整行为可以写成较长函数只要阅读顺序清楚、段落清楚、数据流清楚，提前返回的简单判断优先写成一行紧凑形式减少视觉噪音。函数要么做核心操作逻辑要么做工具函数两者分开写，操作开始时用 print 或 console.log 打印正在做什么操作完成后打印结果让调用者能看到每一步发生了什么。指令函数必须职责单一，一个指令只做一件事，指令之间通过组合而不是嵌套来实现复杂功能，指令不直接操作 UI 只修改数据让数据变化驱动 UI 更新。
代码按自然动作分段，不同语义段之间留空行，功能段落之间空 1 到 2 行，同一语义段中的代码紧凑排列使用尾随注释形成流程整体，相关代码紧靠在一起不相关的代码之间留空行隔开，每个函数之间空 2 行给阅读者视觉喘息空间。复用规则是第一次出现的逻辑优先就地写清楚保持上下文完整可读，同样逻辑稳定出现两次及以上再提取，提取的前提是提取后更容易理解、更容易搜索、更容易修改，提取后的公共内容必须有明确名字和明确用途，提前为可能复用而抽象的函数一律不写等需求真实出现再处理。数据集中存放在 store 文件或文件夹，数据结构直接表达系统真正关心的内容，字段名直接说明含义，store 只负责存储状态和提供读取接口不写业务逻辑，所有修改数据的操作都通过 commands 里的指令函数完成。代码中必须让人快速看出数据从哪里来、经过什么处理、改了什么、最后到哪里去，数据变化优先显式书写避免让人靠隐藏机制去猜，局部代码必须尽量自带足够上下文让人只看这一小段也能知道数据在这里被读、被改、被返回、被影响。
生成代码时固定按这个顺序展开：先给项目结构列出所有文件每个文件一句话说明作用，再给功能说明一句话说这次要实现什么，再给改动位置列出要改哪些文件每个文件改什么，再按文件给出完整代码包含所有注释，最后给使用说明告诉用户怎么运行、怎么测试、以后怎么改。整个生成过程始终围绕事件→指令→数据→反馈这条主线展开，让人能清晰看到交互如何触发事件、事件如何调用指令、指令如何修改数据、数据如何驱动反馈。看项目结构能快速定位主要部分，看目录名和文件名能快速猜出作用，看文件头注释能立刻知道如何使用这个文件，看函数注释能立刻知道这个函数要做什么，看主流程能顺着读懂行为顺序，看局部代码能大致判断怎么改，看代码注释能学到编程技巧和算法思路，同一主体的主要操作集中、连续、可读，数据流动短、直、可追踪，代码阅读过程中不会因为过多碎函数、碎文件、碎跳转而中断理解，生成结果始终优先降低理解成本、追溯成本、修改成本、上下文依赖和心智消耗，始终保持局部可读、局部可改、局部可续写，同时让代码成为最好的编程教材。
